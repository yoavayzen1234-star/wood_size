# מסד נתונים (Supabase / PostgreSQL): ייעול, מחיקות ואינדקסים

מסמך זה מתאר את **מצב הקיים** בפרויקט (מיגרציות + שירות `projects`).  
**עדכון:** מיגרציה `20260410200000_users_rls_and_indexes.sql` מיישמת RLS על `public.users` ואת שני האינדקסים המורכבים המומלצים — להריץ `supabase db push` / להחיל במסד הפריסה.

---

## 1. סכמה רלוונטית (תמצית)

| טבלה | מפתחות / קשרים | הערות |
|------|-----------------|--------|
| `public.users` | `id` → `auth.users(id)` **ON DELETE CASCADE** | שורה ציבורית למשתמש אחרי הרשמה |
| `public.projects` | `id` PK; `user_id` → `users(id)` **ON DELETE CASCADE** | `data` jsonb (legacy), `kerf_mm`, `updated_at` + טריגר |
| `public.project_parts` | `project_id` → `projects(id)` **ON DELETE CASCADE**; **UNIQUE** `(project_id, sort_order)` | מחיקת פרויקט מוחקת שורות אוטומטית |
| `public.project_stock_lengths` | `project_id` → `projects(id)` **ON DELETE CASCADE**; **UNIQUE** `(project_id, material_key, sort_order)` | כנ"ל |

**מסקנה:** מחיקת **פרויקט** ברמת DB מספיקה כדי לנקות חלקים ומלאי — בזכות `ON DELETE CASCADE`, בלי צורך במחיקות נפרדות מהאפליקציה.

---

## 2. האם הכל «יעיל»?

### מה שכבר תומך בביצועים טובים

- **רשימת פרויקטים לפי משתמש:** אינדקס `projects_user_id_idx` על `(user_id)` + (אחרי המיגרציה החדשה) **`projects_user_id_created_at_idx (user_id, created_at)`** — מותאם ל־`getProjects`: `WHERE user_id = ? ORDER BY created_at`.
- **שאילתות לפי פרויקט בטבלאות המשנה:** אינדקסים ייעודיים:
  - `project_parts_project_id_idx` על `(project_id)` + **`project_parts_project_id_client_row_id_idx (project_id, client_row_id)`** למחיקת שורת חלק בודדת
  - `project_stock_lengths_project_id_idx` על `(project_id)`
- **מחיקה / עדכון לפי `project_id`:** השאילתות `DELETE ... WHERE project_id = $1` (ב־RPC ובשמירה הגרנולרית) נשענות על האינדקסים האלה.
- **חיפוש פרויקט לפי `id`:** מפתח ראשי — ללא צורך באינדקס נוסף.
- **מגבלות UNIQUE:** יוצרות אינדקסים משלהן — עוזרות גם לשאילתות שמתאימות למבנה המפתח.

### RLS (Row Level Security)

- על `projects`, `project_parts`, `project_stock_lengths` מופעל RLS עם מדיניות «שייך למשתמש המחובר דרך `projects.user_id`».
- פונקציית `save_project_editor_state` היא **SECURITY DEFINER** ובודקת בעלות לפני שינוי — מתאים לשמירה אטומית בלי לפתוח חור ב־RLS.

### RLS על `public.users` (מיושם במיגרציה `20260410200000`)

- `ENABLE ROW LEVEL SECURITY` על `public.users`.
- פוליסיות ל־**authenticated**: `SELECT`, `INSERT`, `UPDATE` רק כש־`auth.uid() = id`. אין פוליסי `DELETE` מהלקוח (מחיקת חשבון ב־Auth מטפלת ב־cascade).
- טריגר `handle_new_auth_user` נשאר **SECURITY DEFINER** — יצירת השורה אחרי הרשמה לא נחסמת ע״י RLS.

**לאחר פריסה:** ב־Supabase Dashboard → Authentication / Table Editor — לוודא שהמדיניות פעילה וש־`getMyProfile` ממשיך לעבוד (שורה אחת לכל משתמש).

## 3. האם כל המחיקות «קוראות» (מתבצעות) כמו שצריך?

### מחיקת פרויקט שלם

| מקור | מה קורה |
|------|---------|
| **DB** | `DELETE` מ־`projects` → **CASCADE** מוחק `project_parts` ו־`project_stock_lengths`. |
| **אפליקציה** | `deleteProject()` ב־`src/services/projects.ts` — `from('projects').delete().eq('id', id).eq('user_id', userId)` + ניקוי מטמון מקומי (`invalidateProjectState`). |

**כן** — אין צורך בקריאות מחיקה נוספות לילדים; ה־FK מטפל בזה.

### מחיקת שורת חלק בודדת (לפני שמירה מושהית)

| מקור | מה קורה |
|------|---------|
| **אפליקציה** | `deleteProjectPartRow()` — `DELETE` מ־`project_parts` לפי `project_id` + `client_row_id` (כשהמשתמש מוחק שורה בטבלה). |

**כן** — נקרא מה־UI (`useProjectEditor` → `removeRow`).

### מחיקות «מלאות» בתוך שמירה (החלפת תוכן)

| מקור | מה קורה |
|------|---------|
| **RPC** `save_project_editor_state` | `DELETE FROM project_parts WHERE project_id = …` ואז INSERT מחדש; אותו דבר ל־`project_stock_lengths`. |
| **שמירה גרנולרית** (`replaceProjectPartsTable` / `replaceProjectStockTable`) | `DELETE … WHERE project_id = …` ואז INSERT. |

**כן** — אלו לא «מחיקות חסרות» אלא **אסטרטגיית replace** — מתאימה לסנכרון מלא של טבלה מנורמלת אחרי שינוי.

### מה *לא* מבוצע ב-SQL ישיר

- **מחיקת משתמש ב־Auth:** אם משתמש נמחק מ־`auth.users`, ה־CASCADE מוגדר ל־`public.users` ומשם ל־`projects` — היררכיה עקבית. זה לא קורה מהאפליקציה הנוכחית (אין מסך «מחק חשבון» במיפוי הקוד) — אם יתווסף, כדאי לוודא זרימה מול Supabase Auth.

## 4. אינדקסים — מה יש ומה אפשר להוסיף

### קיים במיגרציות

| אובייקט | אינדקס / מגבלה |
|---------|----------------|
| `projects` | `projects_user_id_idx (user_id)` + **`projects_user_id_created_at_idx (user_id, created_at)`** (מיגרציה 20260410200000) |
| `project_parts` | `project_parts_project_id_idx (project_id)` + **`project_parts_project_id_client_row_id_idx (project_id, client_row_id)`** + UNIQUE `(project_id, sort_order)` |
| `project_stock_lengths` | `project_stock_lengths_project_id_idx (project_id)` + UNIQUE `(project_id, material_key, sort_order)` |

### האם «יש אינדקס להכל»?

- **לזרימות העיקריות** — כיסוי מלא לדפוסים המתועדים כאן, כולל מחיקת שורת חלק ורשימת פרויקטים ממוינת.
