# מחשבון חיתוך — עבודה באופליין: ארכיטקטורה ושינויים נדרשים

מסמך זה מתאר **מה צריך לשנות בארכיטקטורה** כדי שהמחשבון (עורך חלקים, חישוב אופטימיזציה, תצוגת תוצאות) **יוכל לפעול בלי חיבור לאינטרנט**, תוך שמירה על התנהגות סבירה כשהרשת חוזרת.

---

## 1. מה נחשב "עובד באופליין"

מומלץ להגדיר רמות ברורות:

| רמה | תיאור |
|-----|--------|
| **א׳ — אפליקציה נטענת** | ה־UI וה־JS זמינים מהמטמון; ניווט ל־SPA לא נכשל. |
| **ב׳ — מחשבון פעיל** | הזנת חלקים, שינוי מלאי, הרצת אופטימיזציה (כולל Workers) — **בלי** קריאות לשרת. |
| **ג׳ — שמירה מקומית** | פרויקטים ומצב עורך נשמרים במכשיר; אין אובדן עבודה בניתוק קצר. |
| **ד׳ — סנכרון עם Supabase** | כשנכנסים שוב לרשת: דחיפה למסד, פתרון סתירות, עדכון רשימת פרויקטים. |

היום האפליקציה ממוקדת ב־**רמה ד׳ כמקור אמת** (Supabase). מעבר לאופליין מלא דורש **מקור אמת מקומי זמני** + **תור סנכרון**.

---

## 2. מצב נוכחי (תלות ברשת)

- **אימות:** `getCurrentUser()` ו־Supabase Auth — דורשים רשת (או session שכבר בזיכרון/אחסון פנימי של ה־SDK; בלי רשת לעיתים session לא מתרענן).
- **נתונים:** פרויקטים, חלקים, מלאי — נטענים ונשמרים דרך Supabase (`getProjects`, `loadProjectEditorState`, `persistProjectEditorIfChanged`, RPC וכו׳).
- **מטמון זיכרון:** `projectCache` — עוזר לביצועים אבל **נמחק** בסגירת הטאב; לא תחליף לאופליין.
- **מטמון localStorage:** `workspaceLocalCache` — פרופיל + רשימת פרויקטים בלבד, TTL קצר; **לא** מכסה עורך מלא ולא מבטיח עקביות אחרי עריכה.
- **Service Worker (`public/sw.js`):** cache-first לנכסים סטטיים מאותו מקור; **לא** מספק offline ל־`index.html` / ניווט, ולא ל־API של Supabase.

**מסקנה:** בלי שינוי ארכיטקטורה, **אין עריכה אמינה ושמירה באופליין**; החישוב עצמו (ספריות מקומיות + Workers) יכול תיאורטית לרוץ אם כבר נטען הקוד — אבל הזרימה סביבו תלויה בנתונים מהשרת.

---

## 3. עקרונות ארכיטקטורה מומלצים

### 3.1 מקור אמת כפול (Local-first)

- **מקומי:** מצב העבודה הנוכחי (פרויקטים, שורות עורך, מלאי, kerf) נשמר ב־**IndexedDB** (או OPFS לקבצים גדולים בעתיד).
- **שרת:** Supabase נשאר מקור האמת **לאחר סנכרון מוצלח**.
- כל שינוי משתמש מסומן כ־**dirty** + נרשם ב־**תור פעולות (outbox)** לשידור כשיש רשת.

### 3.2 שכבת "Repository" אחידה

להפריד את הקוד מהפרטים:

- היום: קריאות ישירות ל־`supabase` מ־`services/projects.ts` וכו׳.
- יעד: ממשק כמו `ProjectRepository` עם מימושים:
  - `RemoteProjectRepository` (ההתנהגות הנוכחית),
  - `LocalProjectRepository` (קריאה/כתיבה ל־IndexedDB),
  - או `HybridRepository` שמחליט לפי `navigator.onLine` + איכות חיבור (אופציונלי).

כך לוגיקת ה־UI (עורך, אופטימיזציה) **לא משתנה עסקית** — רק מקור הנתונים.

### 3.3 סנכרון (Sync engine)

רכיב ייעודי ש:

1. מזהה חזרה לרשת (`online` event + בדיקת Supabase בפועל).
2. דוחף את ה־outbox בסדר בטוח (למשל לפי פרויקט, עם מנעולים).
3. אחרי push מוצלח — מושך שינויים מהשרת (אם ייתכנו עריכות ממכשיר אחר).
4. מטפל בכשלים (retry, backoff, הודעת משתמש).

**Background Sync** (אם נתמך בדפדפן) יכול לעזור לעורר סנכרון כשהמכשיר חוזר לרשת — אבל זה תוספת, לא חובה ל־MVP.

### 3.4 סתירות (Conflict resolution)

מינימום pragmati:

- **Last-write-wins (LWW)** לפי `updated_at` או timestamp מקומי — פשוט, עלול לאבד שינויים במכשיר שני.
- שיפור: **גרסה לפרויקט** (`version` ב־DB) — אם הגרסה המקומית < השרת, למשוך ואז למזג או לבקש מהמשתמש.

מסמך מוצר צריך להחליט אם יש תמיכה ב־**מספר מכשירים** במקביל; זה קובע את רמת המורכבות.

## 4. שינויים לפי שכבות

### 4.1 Service Worker ואפליקציה סטטית (רמה א׳)

- **Precache / App Shell:** בבילד (למשל Workbox עם Vite plugin) — לשמור `index.html`, entry JS, CSS, Workers, אייקונים.
- **Navigation fallback:** בקשות ניווט ל־same-origin מחזירות את ה־`index.html` מהקאש כשאין רשת (כדי שה־SPA יעלה).
- **עדכונים:** אסטרטגיית גרסה לקאש (שם קאש עם build hash) + הודעת "גרסה חדשה זמינה" אופציונלית.

**לא** לקאש בקשות ל־`*.supabase.co` (במיוחד Auth), למעט אולי GET ספציפיים אם בעתיד יוגדרו במפורש — בדרך כלל לא רלוונטי.

### 4.2 אימות (רמה ב׳–ג׳)

אפשרויות (לא סותרות):

1. **משתמש כבר מחובר:** Supabase שומר session ב־localStorage/IndexedDB — **פתיחת האפליקציה באופליין** עשויה עדיין לקרוא session מקומית ולהציג מצב "מחובר" **בלי** אימות חדש. צריך לבדוק בפועל את גרסת ה־SDK והגדרות `persistSession`.
2. **כניסה ראשונה / שחזור session באופליין:** בלי רשת — **לא** ניתן להתחבר עם email/sסיסמה סטנדרטי. אפשרויות מוצר:
   - חסימת מסך עם "נדרש חיבור לכניסה".
   - **מצב אורח מקומי** (ללא משתמש Supabase) — כל הנתונים רק במכשיר; אופציונלי לקשר לחשבון מאוחר יותר (מורכב).
3. **Token refresh:** רענון JWT דורש רשת; באופליין ארוך — הפעולות צריכות להישאר מקומיות עד שיש רשת.

### 4.3 אחסון נתונים מקומי (רמה ג׳)

- **IndexedDB** עם סכמה ברורה: טבלאות/אוספים לפרויקטים, snapshot עורך, outbox.
- **מיגרציות סכמה מקומיות** (גרסת DB מקומית) — כמו migrations בשרת, כדי שלא יישברו עדכוני אפליקציה.
- שימוש חוזר בפורמט `ProjectEditorPayload` הקיים כדי **לא לשנות** את מודל העסקים — רק איפה נשמר.

### 4.4 אופטימיזציה ו־PDF (רמה ב׳)

- החישוב (Workers + `cuttingOptimizer`) כבר רץ בדפדפן — **אין צורך בשרת** לחישוב עצמו.
- ייצוא PDF: תלוי ב־`jspdf` וכו׳ — אם הקוד כבר בקאש, **יכול** לעבוד באופליין; יש לוודא שכל ה־chunks נטענים ב־precache או נטענים לפני הניתוק.

### 4.5 סנכרון עם Supabase (רמה ד׳)

- מיפוי פעולות: יצירת פרויקט, מחיקה, שינוי שם, שמירת עורך — כל אחת **פריט outbox** עם payload ו־idempotency key (למניעת כפילויות).
- אחרי סנכרון: עדכון `workspaceLocalCache` / מטמון הפרויקטים כדי שכניסה הבאה תהיה עקבית.

## 5. סיכונים ומגבלות

- **אובדן נתונים:** אם המשתמש מנקה אתר נתונים של האתר — IndexedDB נמחק; צריך להבהיר ב־UX.
- **מכשירים מרובים:** LWW בלבד עלול למחוק עבודה; נדרשת מדיניות מוצר.
- **אבטחה:** נתונים רגישים במכשיל — כמו היום ב־localStorage לטוקנים; לא חריג, אבל כדאי לתעד.
- **בדיקות:** נדרשים בדיקות e2e לתרחישים: offline → עריכה → online → סנכרון; שני טאבים; רענון באמצע.

---

## 6. שלבי יישום מומלצים (סדר עבודה)

1. **SW + App Shell** — גלישה באופליין אחרי ביקור אחד מקוון (טעינת האפליקציה).
2. **IndexedDB + שכבת repository** — כתיבה/קריאה מקומית לפרויקט פעיל אחד (בלי סנכרון מלא).
3. **מצב offline מפורש ב־UI** — באנר "אין רשת; השינויים יישמרו במכשיר".
4. **Outbox + סנכרון חד-כיווני** (מקומי → שרת) למשתמש שמחובר ועובד על מכשיר יחיד.
5. **משיכה מהשרת אחרי סנכרון** + מדיניות סתירות.
6. (אופציונלי) Background Sync, מצב אורח, ייצוא/ייבוא גיבוי קובץ.

## 7. קבצים/אזורים בקוד שיושפעו (התחלה לרשימת משימות)

| אזור | למה |
|------|-----|
| `public/sw.js` או מעבר ל־Workbox | precache, navigation fallback |
| `src/App.tsx` / זרימת auth | זיהוי offline, מסלול "מקומי בלבד" או חסימה |
| `src/services/projects.ts` | פיצול או עטיפה ל־repository היברידי |
| `src/hooks/useProjectEditor.ts` | שמירה ל־IndexedDB + סימון dirty |
| `src/lib/projectCache.ts` | אינטגרציה או החלפה חלקית במטמון persistent |
| מודול חדש: `src/sync/` | outbox, sync loop, retry |
| סכמת IndexedDB חדשה | גרסאות, אינדקסים לפי `projectId` |

## 8. סיכום

כדי שהמחשבון **יעבוד באופליין** בצורה אמינה, לא מספיק לשפר קאש ל־JS — צריך **מקור אמת מקומי** (IndexedDB), **שכבת גישה לנתונים** מופרדת מהרשת, ו־**מנגנון סנכרון** עם Supabase כשחוזרים לרשת. ה־Service Worker משמש בעיקר ל־**זמינות האפליקציה והנכסים**, והלוגיקה העסקית של שמירה וסנכרון נשארת ב־**main thread / worker ייעודי לסנכרון**, לא ב־SW בלבד.

מסמך זה הוא **מפת דרכים ארכיטקטונית**; פירוט סכמות DB בשרת (אם נוסיף עמודות `version` וכו׳) יתווסף בשלב תכנון מפורט נפרד.
