# טעינה מהירה של האתר — הסבר ומפת דרכים

מסמך זה מסביר **מה קורה ברגע שנכנסים לאתר**, למה לעיתים זה מרגיש איטי, **מה כבר מיושם** בפרויקט כדי לזרז, ומה אפשר לעשות בהמשך כדי שהממשק יופיע ויעבוד מהר יותר.

---

## 1. מה קורה בפועל בכניסה לאתר (סדר כרונולוגי)

### שלב א׳ — הדפדפן מוריד את האפליקציה

1. נטען `index.html`.
2. נטען ה־**bundle הראשי** של Vite/React (`main.tsx` וכל מה שתלוי בו ישירות).
3. רק אחרי ש־JavaScript רץ — React מצייר את המסך.

**למה זה חשוב:** עד שהקובץ הראשון מספיק גדול נטען ומפוענח, המשתמש רואה מסך ריק או “טוען…” זמן מה.

### שלב ב׳ — Service Worker (אופציונלי)

ב־`main.tsx` נרשם Service Worker מינימלי (`/sw.js`) אחרי אירוע `load`.

- **בכניסה ראשונה:** בדרך כלל אין עדיין מטמון — הטעינה דומה ל“אתר רגיל”.
- **בכניסות חוזרות:** אם ה־SW מגדיר קאש לקבצים סטטיים, הורדת ה־JS/CSS יכולה להיות מהירה יותר — תלוי במימוש ב־`public/sw.js`.

### שלב ג׳ — אימות והכנת “סביבת עבודה” (משתמש מחובר)

ב־`App.tsx`, אחרי עליית האפליקציה:

1. קריאה ל־`getCurrentUser()` (Supabase) — **רשת**.
2. אם יש משתמש — נטען דינמית `preloadUserData.ts` ואז רצה `preloadUserWorkspaceData()`:
   - **במקביל:** `getMyProfile()` + `getProjects()` (`Promise.all`).
   - אחר כך: טעינה **חוסמת** של מצב העורך לפרויקט “הנוכחי” (האחרון מ־`localStorage` או הראשון ברשימה) — `loadProjectEditorState(...)`.
   - **ברקע (לא חוסם):** מתחילה טעינת שאר הפרויקטים למטמון.

כל עוד `authLoading === true`, המשתמש רואה מצב טעינה לפני שהדף המלא נפתח.

### שלב ד׳ — דפים כבדים (Code splitting)

`LoginPage` ו־`AuthedCalculatorPage` נטענים ב־`lazy()`.

- בפעם הראשונה שנכנסים למסך המחושב — הדפדפן מוריד **chunk נוסף**.
- אם יש בתוך העמוד ייבואים כבדים (למשל PDF, ספריות גדולות), הם עשויים להיטען **רק כשפותחים פיצ’ר** — או אם הם עדיין חלק מה־chunk של העמוד.

## 2. למה זה “מרגיש איטי” (גורמים נפוצים)

| גורם | הסבר קצר |
|------|-----------|
| **גודל ה־bundle הראשוני** | ככל שיש יותר קוד בנתיב הקריטי של הכניסה — יותר הורדה ופענוח לפני UI אמיתי. |
| **מספר קריאות רשת ל־Supabase** | פרופיל + רשימת פרויקטים + מצב עורך לפרויקט אחד = לפחות כמה round-trips. רשת איטית / latency גבוהה = המתנה מורגשת. |
| **טעינה חוסמת לפני הצגת UI** | כל עוד `authLoading` נשאר true עד סיום ה־preload, המשתמש מחכה גם אם חלק מהנתונים לא חיוניים למסך הראשון. |
| **Chunk ראשון של העמוד המאומת** | `AuthedCalculatorPage` יכול לכלול הרבה UI ולוגיקה; הורדת ה־chunk השני מוסיפה עיכוב אחרי שכבר “התחברנו”. |
| **מכשיר חלש / דפדפן** | פענוח JS ורינדור React על מובייל יכולים להאט משמעותית. |

---

## 3. מה כבר קיים בפרויקט (נקודות חוזק)

- **טעינת פרויקטים “חכמה”:** אחרי טעינת הפרויקט הפעיל, שאר הפרויקטים נטענים **במקביל ברקע** (`preloadUserWorkspaceData`) — לא אמור לחסום את המשתמש לנצח.
- **פיצול דפים:** `LoginPage` / `AuthedCalculatorPage` ב־`lazy` — מפחית את גודל הטעינה הראשונית לעומת טעינת הכל בבת אחת.
- **ביטול בקשות:** שימוש ב־`AbortController` כדי לא לעדכן state אחרי יציאה מהרכיב — מונע עבודה מיותרת.

## 4. מה לעשות כדי שהכניסה תרגיש מהירה יותר (המלצות מעשיות)

הרשימה מסודרת לפי **השפעה לעומת מאמץ** (בערך).

### א. מדידה לפני שמשנים (חובה)

- **Chrome DevTools → Network:** סינון `JS`, בדיקת גודל וזמן טעינה של ה־entry chunk ושל chunk של `AuthedCalculatorPage`.
- **Lighthouse (Performance):** זמן FCP/LCP, “Reduce JavaScript execution time”.
- **Vite analyze:** `npm run build` במצב analyze אם מוגדר (`rollup-plugin-visualizer` ב־`vite.config.ts`) — רואים מה “שוקל” בבנדל.

בלי מדידה קל לשפר את הדבר הלא נכון.

### ב. להציג UI מהר יותר — “Progressive loading”

רעיון: **לא לחכות** לסיום כל `preloadUserWorkspaceData` לפני שמציגים את המעטפת של האפליקציה (סkeleton / כותרת / פריסה).

- להוריד את `authLoading` מוקדם יותר אחרי `getCurrentUser()` בלבד.
- להציג את `AuthedCalculatorPage` עם מצבי “טוען פרויקטים…” בתוך העמוד, בזמן שהנתונים מגיעים.

**תוצאה:** המשתמש מרגיש שהאתר “עלה”, גם אם הנתונים עדיין נטענים.

### ג. להקטין את ה־bundle הראשוני

- **Dynamic import** לפיצ’רים כבדים (PDF, דוחות) — רק בלחיצה על “ייצוא”.
- לוודא שאין ייבוא “מקרי” של מודולים כבדים בקבצים שנטענים בכניסה.
- לבדוק אם `@supabase/supabase-js` נטען מוקדם מדי; לעיתים אפשר לדחות חלק מהאתחול.

### ד. לצמצם round-trips ל־Supabase

- **איחוד נתונים:** אם אפשר, endpoint אחד / view אחד שמחזיר “סיכום workspace” במקום 2–3 קריאות נפרדות.
- **אינדקסים ו־RLS:** שאילתות מהירות = פחות המתנה בצד השרת (כבר קיימות מיגרציות — לוודא שהן deployed).

### ה. שיפור חוויית מטמון (CDN / SW)

- לוודא שהאתר מוגש עם **כותרות Cache-Control** נכונות לקבצים עם hash (מ־Vite build).
- להרחיב את `sw.js` (בזהירות) לקאש של `assets/*.js` אחרי התקנה — כדי שכניסה חוזרת תהיה כמעט מיידית.

### ו. חוויית משתמש בזמן המתנה

- Skeleton שמשקף את הפריסה האמיתית (לא רק “טוען…” גנרי).
- `Suspense` עם fallback עקבי.

## 6. סיכום

האיטיות בכניסה נובעת בדרך כלל משילוב: **הורדת JavaScript**, **מספר קריאות רשת**, ו־**מדיניות UI שמחכה לכל הנתונים לפני הצגה מלאה**. בפרויקט כבר יש פיצול דפים וטעינת פרויקטים ברקע — הצעד עם **השפעה גבוהה ביותר** לרוב הוא **להציג את הממשק מוקדם יותר** ולטעון נתונים בשכבות, יחד עם **הקטנת bundle** של הנתיב הראשון.

אם תרצה, אפשר בשלב הבא לבחור יחד 1–2 שינויים קונקרטיים בקוד (למשל: פיצול נוסף או שינוי זרימת `authLoading`) וליישם אותם עם מדידה.
